# Data Sources and Handling — WildfiresAI

WildfiresAI is designed to integrate multiple scientific data sources in order to support structured analysis of wildfire-exposed environments and related infrastructure.

The repository contains pipeline logic only. Large datasets and external data products are treated as runtime inputs and are not versioned in the repository.

---

# Overview

WildfiresAI is designed to combine several categories of scientific data:

- Satellite-derived fire detections
- Topographic information
- Meteorological and environmental context
- Materials and physical property data (via Materials Project)

At present, the core implemented analysis pipeline focuses on materials properties retrieved from the Materials Project database. Environmental and wildfire datasets are supported as optional or future integrations within the architecture.

The repository contains only the code and pipelines required to access and analyze these data sources. Raw datasets are downloaded dynamically or generated locally and excluded from version control.

---

# Satellite Fire Data (NASA FIRMS)

WildfiresAI is designed to integrate active fire detections from NASA FIRMS (Fire Information for Resource Management System).

Near–real-time satellite products that can be used include:

- VIIRS Suomi-NPP (NRT)
- MODIS (NRT)

These satellite products detect thermal anomalies corresponding to active fires.

Fire data can be retrieved through the FIRMS API and parameterized using:

- geographic bounding boxes (regional or global)
- time windows (for example 24 hours or 7 days)

Environment variables generated by `setup_firms_env.sh` define the FIRMS API endpoints used by the system.

The FIRMS API key (`FIRMS_MAP_KEY`) must be provided locally but is never committed to the repository.

These integrations are optional and may be activated depending on the analysis configuration.

---

# Topography and Elevation Data

WildfiresAI supports integration of terrain data through Digital Elevation Model (DEM) datasets.

DEM tiles may be used to derive:

- slope
- altitude
- terrain exposure

DEM files are typically large (approximately 50 MB per tile). For this reason:

- they are stored locally under `data/raw/`
- they are excluded from version control via `.gitignore`

These datasets can be re-downloaded when required.

---

# Meteorological and Environmental Data

The architecture is designed to integrate weather and environmental datasets from external providers.

Potential sources include:

- ECMWF
- NOAA
- Copernicus / EFFIS
- National and regional wildfire services such as NIFC

Corresponding API endpoints and credentials are configured via environment variables.

These integrations are optional and activated depending on the analysis scope.

---

# Materials and Physical Properties

WildfiresAI interfaces with the Materials Project API to retrieve material-level physical properties.

Typical retrieved fields include:

- band_gap
- energy_above_hull
- density
- symmetry
- bulk modulus
- shear modulus

These physical properties are used by the core analysis pipeline to evaluate material suitability under defined constraints.

Access to the Materials Project API is controlled via the environment variable:

```
MP_API_KEY
```

The API key is stored locally in the user's `.env` file and is never committed to the repository.

---

# Data Management Policy

WildfiresAI follows strict rules for handling external data.

The repository does not commit:

- API keys or credentials
- large binary data files
- machine-specific configuration

All sensitive configuration is handled through local environment variables.

Typical setup:

```
.env
.env.example
```

The `.env.example` file documents required variables without exposing credentials.

This approach ensures:

- reproducibility
- security
- clean separation between code and data

---

# Reproducibility

All data used by the project can be reproduced through:

- the provided scripts
- public APIs
- documented environment variables

Because the pipeline stores intermediate artifacts such as analysis results and plots, each execution can be inspected and reproduced without relying on hidden model state.

The repository remains lightweight while preserving full scientific transparency.
